# r2d2

**Repository:** https://github.com/jcsda-internal/r2d2
**Branch:** develop
**Role in JEDI workflow:** the data archive — a MySQL-backed catalog of inputs and outputs (backgrounds, observations, analyses, diagnostics) shared across experiments and platforms.

## What it is

R2D2 = **Research Repository for Data and Diagnostics**. It is a Python
client + MySQL backend that EWOK and Skylab use to fetch, store, and
locate data files across local disks, HPC scratch, and AWS S3. The
README captures it crisply: R2D2 implements CRUD operations on two
classes — `R2D2Index` (metadata-only records) and `R2D2Data` (records
that point at actual files on disk or in cloud storage).

## How it fits into the workflow stack

- **EWOK** uses R2D2 inside its task scripts to fetch backgrounds and
  observations and to register experiment outputs.
- **Skylab** experiments reference R2D2-managed data by metadata
  (model, experiment id, date, resolution, etc.) rather than by path.
- **simobs** reads/writes feedback files via R2D2 in production setups.

## Key concepts

**`R2D2Index`** items (metadata only):
compiler, compute_host, data_hub, data_store, experiment,
file_extension, item, lifetime, model, observation_type, provider,
queue, user.

**`R2D2Data`** items (file-backed):
analysis, bias_correction, diagnostic, feedback, forecast, media,
observation.

**Lifetime tiers** (from the README):
`debug` (14 d), `science` (180 d), `publication` (1825 d),
`release` (indefinite).

**Data store types**: archive, experiments, gfsensemble, skylab,
geosensemble, 4denvar, mpasensemble.

## Key directories

- `src/r2d2/` — the Python client library (`from r2d2 import
  R2D2Index, R2D2Data`).
- `server/` — server-side components; `server/app/` is the web app,
  `server/cfn/` holds CloudFormation, `server/docker/` the container
  spec, `server/scripts/` admin scripts.
- `test/` — pytest suite.

## Key entry points

- `from r2d2 import R2D2Index` and `from r2d2 import R2D2Data` — the two
  user-facing classes.
- `R2D2Index.register(...)`, `R2D2Index.get(...)`, `R2D2Index.search(...)`
- `R2D2Data.fetch(...)`, `R2D2Data.store(...)`, `R2D2Data.search(...)`,
  `R2D2Data.find_by_index(...)`
- `setup.py` — installs the client.

The README has worked examples for every common method. When in doubt,
read it — it doubles as the API reference.

## Common tasks

- **Fetch an analysis file** — `R2D2Data.fetch(item='analysis', model=
  'mom6', experiment='7fhe54', date=..., target_file=...)`.
- **Store a new observation** — `R2D2Data.store(item='observation',
  provider=..., observation_type=..., source_file=...)`.
- **Find which data stores hold a file** —
  `R2D2Data.find_by_index(item='observation', index=2172)`.
- **Delete an experiment's data** —
  `R2D2Index.delete_experiment(name='401b91')` (subject to lifetime
  rules).

## Gotchas

- R2D2 enforces lifetime: experiments can't be deleted before
  `lifetime_days` have elapsed unless `ignore_lifetime=True` is passed
  by an admin.
- `delete_experiment` must run on the same `compute_host` the
  experiment was registered on.
- The `member` keyword on analysis/feedback/forecast/media items
  implies an ensemble member — use it correctly to avoid clobbering
  deterministic data.
- Required keys per item are listed in the README's "Required Keys"
  tables; missing a required key produces unhelpful errors.
