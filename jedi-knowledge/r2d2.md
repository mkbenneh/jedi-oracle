# r2d2

**Repository:** https://github.com/jcsda-internal/r2d2
**Branch tracked:** develop
**Role in JEDI:** the data archive — a MySQL-backed catalog of inputs and outputs (backgrounds, observations, analyses, diagnostics) shared across experiments and platforms.

## What it is

R2D2 = **Research Repository for Data and Diagnostics** (`setup.py`
version 3.0.0, Python ≥3.9). It is a Python client + MySQL backend (plus a
REST server under `server/`) that EWOK and Skylab use to fetch, store, and
locate data files across local disks, HPC scratch, and AWS S3. R2D2
implements CRUD operations on two classes — `R2D2Index` (metadata-only
records) and `R2D2Data` (records that point at actual files on disk or in
cloud storage).

## How it fits into the bundle

- **EWOK** registers every experiment at creation time
  (`ewok/src/ewok/r2d2_util.py` → `R2D2Util.register_experiment`, default
  lifetime `debug`) and uses R2D2 inside its runtime tasks to fetch
  backgrounds/observations and register outputs.
- **Skylab** experiments reference R2D2-managed data by metadata
  (model, experiment id, date, resolution, etc.) rather than by path.
- **simobs**-generated plots are stored as R2D2 `media` items when EWOK
  drives the plotting tasks.

## Key concepts

**`R2D2Index`** items (metadata only):
compiler, compute_host, data_hub, data_store, experiment,
file_extension, item, lifetime, model, observation_type, provider,
queue, user.

**`R2D2Data`** items (file-backed):
analysis, bias_correction, diagnostic, feedback, forecast, media,
observation.

**Lifetime tiers** (from the README):
`debug` (14 d), `science` (180 d), `publication` (1825 d),
`release` (indefinite).

**Reserved data_store_type values**: archive, experiments, gfsensemble,
skylab, geosensemble, 4denvar, mpasensemble.

Every item is identified by its `item` value plus either a `name`
(R2D2Index) or the MySQL `index` integer.

## Key directories

- `src/r2d2/` — the Python client library:
  - `r2d2_index.py` / `r2d2_data.py` — the two user-facing classes
    (`from r2d2 import R2D2Index, R2D2Data`).
  - `r2d2_item.py`, `r2d2_object.py` — base/item plumbing.
  - `r2d2_data_register.py`, `r2d2_compute_host_register.py` —
    registration helpers.
  - `mysql/` — `database_connector.py` (connection from `MYSQL_*` env
    vars), `query_processor.py`, `schema_property.py`.
  - `error/` — typed exceptions (`RegistrationNotFound`,
    `DuplicateRecordFound`, `DataStorePermission`, `MissingKey`, …).
  - `util/` — `cloud_util.py` (incl. `get_s3_data_store`, also exported
    from the package root), `date_util.py`, `logging_util.py`,
    `error_util.py`.
- `server/` — the REST server: `server/app/` (`app.py`, `routes.py`,
  `auth.py`, `swagger.py`, `generator.py`, `gunicorn.py`), `server/cfn/`
  CloudFormation, `server/docker/` container spec, `server/scripts/`
  admin scripts, own `setup.py` and `app.yaml`. `server/README.md`
  documents running a local server + MySQL via docker compose.
- `test/` — pytest suite (`test_query_processor.py`,
  `test_initialize.py`, …) plus schema setup scripts
  (`set_r2d2_test_schema.sh`).

## Configuration / environment

The client reads its MySQL connection from environment variables:
`MYSQL_USER` (default `r2d2`), `MYSQL_DATABASE` (default `r2d2`),
`MYSQL_HOST`, `MYSQL_PORT`, `MYSQL_PASSWORD`
(see `src/r2d2/mysql/database_connector.py`). Logging is controlled by
`R2D2_LOG_LEVEL` / `R2D2_LOG_FILE`; publishing uses
`R2D2_S3_PUBLISH_BUCKET`.

## Key entry points

- `from r2d2 import R2D2Index, R2D2Data` — the two user-facing classes.
- `R2D2Index`: `register(...)`, `get(...)`, `get_by_index(...)`,
  `get_index_by_name(...)`, `get_name_by_index(...)`, `search(...)`,
  `update(...)`, `delete(...)`, `delete_experiment_by_name(...)`.
- `R2D2Data`: `fetch(...)`, `store(...)`, `search(...)`,
  `find_by_index(...)`, `get_by_index(...)`,
  `exists_in_any_data_store(...)`, `update(...)`, `delete(...)`.

The README has worked examples for every common method, plus the
"Required Keys" and "Reserved Values" lookup tables. When in doubt, read
it — it doubles as the API reference.

## Common tasks

- **Fetch an analysis file** — `R2D2Data.fetch(item='analysis', model=
  'mom6', experiment='7fhe54', date=..., target_file=...)`.
- **Store a new observation** — `R2D2Data.store(item='observation',
  provider=..., observation_type=..., source_file=...)`.
- **Find which data stores hold a file** —
  `R2D2Data.find_by_index(item='observation', index=2172)`.
- **Delete an experiment's data** —
  `R2D2Index.delete_experiment_by_name(name='401b91')` (subject to
  lifetime rules; optional `ignore_lifetime=True` for admins).

## Gotchas

- R2D2 enforces lifetime: experiments can't be deleted before
  `lifetime_days` have elapsed unless `ignore_lifetime=True` is passed,
  and `release`-lifetime experiments can never be deleted this way.
- `delete_experiment_by_name` must run on the same `compute_host` the
  experiment was registered on, by the owning user or an admin.
- The `member` keyword on analysis/feedback/forecast/media items
  implies an ensemble member — use it correctly to avoid clobbering
  deterministic data.
- Required keys per item are listed in the README's "Required Keys"
  tables; missing a required key produces unhelpful errors.
- `file_extension`, `observation_type`, and `provider` index items are
  auto-registered; other index items need explicit `register` calls.
- **Delete-by-index removed (2026, r2d2#830/#767):** `delete_by_index`
  was deprecated/removed in favor of attribute-based `delete`/`update`;
  `search` results no longer include the item index (r2d2#836). Old
  scripts that round-trip indexes from search output need updating.
- **File metadata now removed from DB when data files are deleted
  (2026-05):** `src/r2d2/r2d2_data.py` updated (r2d2#843). Previously,
  deleting a data file left orphaned DB metadata rows; this is now
  cleaned up automatically. Good news for housekeeping, but code that
  relied on deleted files still having DB entries will break.
